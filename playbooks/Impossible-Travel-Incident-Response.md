# Security Operations Center Playbook

## Incident Type

Impossible Travel / Geographic Anomaly Authentication

------------------------------------------------------------------------

# 1. Objective

This playbook provides SOC analysts with a standardized procedure to
investigate and respond to **Impossible Travel authentication alerts**.

Objectives:

-   Detect potential credential compromise
-   Identify unauthorized account access
-   Investigate suspicious geographic login patterns
-   Contain compromised accounts
-   Prevent account takeover and lateral movement

------------------------------------------------------------------------

# 2. Scope

This playbook applies to authentication systems including:

-   Microsoft 365 / Azure AD
-   Okta
-   Google Workspace
-   VPN authentication portals
-   Cloud identity providers
-   Enterprise SaaS platforms

Alerts are typically generated by:

-   Identity Protection systems
-   SIEM correlation rules
-   Cloud authentication monitoring tools
-   User behavior analytics platforms

------------------------------------------------------------------------

# 3. Threat Description

Impossible Travel alerts occur when a user account logs in from **two
geographically distant locations within an impossible timeframe**.

Example scenario:

  Time    Location
  ------- ---------------
  09:00   New York, USA
  09:35   Singapore

Since it is physically impossible to travel between these locations in
such a short period, the alert indicates potential:

-   Credential compromise
-   VPN misuse
-   Session token theft
-   Account takeover

Attackers commonly obtain credentials through:

-   Phishing attacks
-   Password spraying
-   Credential stuffing
-   Malware stealing authentication tokens

------------------------------------------------------------------------

# 4. Detection Sources

Primary detection sources include:

-   Azure AD Sign-In Logs
-   Okta Authentication Logs
-   VPN Authentication Logs
-   Identity Protection Alerts
-   SIEM Correlation Rules

Key indicators include:

-   Multiple login locations in short timeframe
-   Logins from high-risk countries
-   Authentication from anonymizing networks (VPN/Tor)
-   Login behavior inconsistent with normal user activity

------------------------------------------------------------------------

# 5. Alert Triage

When an Impossible Travel alert is generated, the SOC analyst should
collect:

-   Username
-   Source IP addresses
-   Geographic locations of logins
-   Login timestamps
-   Authentication method used
-   Device identifiers

Example alert:

Alert: Impossible Travel Detected\
User: john.doe@company.com\
Login 1: New York, USA -- 08:55 AM\
Login 2: Germany -- 09:20 AM

Initial triage questions:

-   Does the user normally travel internationally?
-   Were both logins successful?
-   Was MFA used during authentication?
-   Are either login IP addresses known VPN providers?

------------------------------------------------------------------------

# 6. Investigation Procedure

## Step 1 --- Verify Login Locations

Review authentication logs to confirm the login locations.

Example SIEM query:

    user = "target_user"
    | stats count by source_ip, location

Determine:

-   Origin country of login attempts
-   Whether IP addresses belong to VPN providers
-   Whether login occurred from corporate network

------------------------------------------------------------------------

## Step 2 --- Review User Activity Timeline

Reconstruct the authentication timeline:

-   First login timestamp
-   Second login timestamp
-   Duration between sessions

Evaluate whether travel between locations is physically possible.

------------------------------------------------------------------------

## Step 3 --- Identify Suspicious Indicators

Indicators of compromise include:

-   Login from previously unseen location
-   Login from high-risk country
-   Login from anonymized IP infrastructure
-   Login outside normal work hours

Also check:

-   User agent strings
-   Device identifiers
-   Browser fingerprint changes

------------------------------------------------------------------------

## Step 4 --- Investigate Post-Login Activity

If authentication succeeded, review account activity.

Investigate:

-   File downloads
-   Email access
-   Cloud console activity
-   Administrative actions

Check for suspicious commands or actions such as:

    Creation of mailbox forwarding rules
    Mass file downloads
    Privilege escalation attempts
    API token creation

------------------------------------------------------------------------

# 7. Containment

If account compromise is suspected:

Immediate actions:

-   Force password reset
-   Revoke active authentication sessions
-   Revoke authentication tokens
-   Temporarily disable account if necessary

Block suspicious IP addresses if malicious activity is confirmed.

------------------------------------------------------------------------

# 8. Eradication

Remove attacker access.

Actions include:

-   Reset credentials
-   Remove malicious OAuth or API tokens
-   Remove unauthorized mailbox rules
-   Audit privileged access granted during compromise

------------------------------------------------------------------------

# 9. Recovery

Restore normal account access.

Recommended improvements:

-   Enforce Multi-Factor Authentication (MFA)
-   Implement conditional access policies
-   Restrict logins from high-risk countries
-   Deploy identity risk detection controls

Continue monitoring authentication logs for suspicious activity.

------------------------------------------------------------------------

# 10. Post-Incident Activities

Conduct incident review.

Tasks include:

-   Root cause analysis
-   User awareness training if phishing involved
-   Detection rule tuning
-   Identity monitoring improvements

Lessons learned should be incorporated into SOC playbooks.

------------------------------------------------------------------------

# 11. Roles and Responsibilities

SOC Analyst - Investigate authentication anomalies - Determine risk of
compromise - Collect forensic evidence

Incident Response Team - Contain compromised accounts - Coordinate
remediation actions

Security Engineering - Improve identity monitoring rules - Deploy
stronger authentication controls
